# Y-Agent 项目愿景

## 项目定位

Y-Agent 是一个面向开发者的 Agent Harness，而不是通用聊天机器人，也不试图把某个模型包装成 AGI。

它与 Codex 属于同一类系统：在 LLM 外围提供目标管理、上下文、工具、计划、委派、权限、恢复和可观测性，使模型能够完成真实、长期、可审计的任务。Y-Agent 的差异化方向是 Rust 优先、模型无关、自编排和自进化。

## 核心判断

LLM 会快速变化，但 Harness 的基础职责相对稳定：

- 理解并持续保存用户目标与约束
- 在 fast、plan、loop 等执行模式之间选择
- 组织上下文、知识、Skills 和工具
- 将复杂任务拆解为计划、工作流或子 Agent
- 在高风险操作前请求授权
- 在失败、中断或进程重启后恢复
- 记录完整执行轨迹，并把有效经验沉淀为可复用能力

Y-Agent 的长期价值不在于绑定最强模型，而在于让不同模型都能在清晰的系统边界内稳定工作。

## 核心能力

### 1. 目标驱动

目标、约束、决策和进度应贯穿整个任务生命周期，而不是只存在于第一条用户消息中。

当前目标语义由 Working Memory 和 Handoff 文档承载；未来可以在确有跨会话生命周期需求时，再演进为独立的持久化 Goal 对象。

### 2. Plan 与 Loop

- Plan 模式适合边界清晰、可提前拆解的任务，支持计划审阅、阶段依赖、并行执行和恢复。
- Loop 模式适合探索性任务，通过执行、自检和下一轮决策逐步收敛。
- Fast 模式处理无需额外编排的简单任务。

模式是执行策略，不应复制业务逻辑。

### 3. 自编排

Agent 可以根据任务动态选择工具、委派子 Agent、创建工作流并执行 DAG。可复用流程应沉淀为工作流或 Skill，而不是长期依赖一次性的超长提示词。

### 4. 自进化

系统从已完成任务中捕获经验，提取稳定模式，生成 Skill 改进建议，并通过回归检查和人工批准控制演进风险。

自进化不是无约束地修改自己，而是一个可追踪、可比较、可回滚的工程流程。

### 5. 知识与 Skills

- 知识库保存外部事实和领域材料，支持多粒度切分与混合检索。
- Skills 保存可复用的推理和操作方法，采用小根文档与按需加载结构。
- Tools 负责确定性的外部操作，不应与 Skills 混为一体。

### 6. 安全与恢复

安全由多层机制共同承担：运行时隔离、Guardrails 拦截、权限规则和 HITL 审批。恢复能力由 WAL、检查点、会话记录、文件 Journal 和 Rewind 共同提供。

### 7. 可观测性与 OTel 生态

每次模型调用、工具执行、子 Agent 委派、成本和失败都应可追踪。

当前实现提供本地 diagnostics 和可选 Langfuse 原生导出。通用 OpenTelemetry exporter 是明确的扩展方向；在代码真正接入 OTel SDK/exporter 前，文档不应把它描述为已完成能力。

## 工程原则

1. 模型无关：Provider 可以替换，Harness 语义保持稳定。
2. Service 层拥有业务逻辑：CLI、Web、GUI 只负责输入输出。
3. 显式状态：目标、计划、权限、检查点和错误都可检查。
4. 可恢复优先：失败后从最近可靠状态继续，而不是从头猜测。
5. Token 效率：按需加载知识、Skills 和工具描述。
6. 可验证演进：能力升级必须有测试、指标、版本和回滚路径。
7. 诚实记录成熟度：愿景、实验和已实现功能必须清楚区分。

## 非目标

- 不追求无需配置即可解决所有问题。
- 不把某个模型厂商的专有行为固化为核心架构。
- 不用大量历史设计稿代替当前代码和测试。
- 不把一次成功的 Agent 行为直接视为稳定能力。

## 成功标准

Y-Agent 成功时，开发者可以选择任意合适模型，为任务提供知识、Skills 和工具，然后让 Harness 以可控、可恢复、可观测的方式持续完成目标，并将验证过的经验转化为下一次任务的能力。
